# Security Audit スキル

black-hacker と white-hacker の 2 エージェントを連携させ、プロジェクトのセキュリティを網羅的に診断する。

---

## 実行フロー

### Phase 0: スコープ確認

診断対象が不明確な場合は確認する。明確なら即 Phase 1 に進む。

- 対象ディレクトリ / ファイル（デフォルト: プロジェクトルート全体）
- 特に重点的に見たい箇所（認証・API・DB など）があるか

### Phase 1: black-hacker による攻撃分析

black-hacker エージェントに以下の指示で委任する：

```
以下のスコープに対して攻撃分析を実施してください。
対象: [スコープ]
重点箇所: [あれば記載]

出力は「攻撃シナリオレポート」フォーマット（エージェントのシステムプロンプト準拠）で、
特に「white-hacker への引き継ぎメモ」セクションを含めてください。
```

black-hacker の出力全体を変数として保持する（Phase 2 に渡すため）。

### Phase 2: white-hacker による防御立案

white-hacker エージェントに以下の指示で委任する：

```
black-hacker から以下の攻撃シナリオレポートを受け取りました。
各脅威に対する具体的な対策を立案・提案してください。

--- black-hacker レポート ---
[Phase 1 の出力をそのまま貼り付ける]
---

出力は「対策レポート」フォーマット（エージェントのシステムプロンプト準拠）で、
優先度 P0（即時対応）から順に記載してください。
```

### Phase 3: 統合レポート生成

両エージェントの出力を統合し、`SECURITY_REPORT.md` としてプロジェクトルートに保存する。

---

## SECURITY_REPORT.md テンプレート

必ずこのテンプレートに従う：

```markdown
# Security Audit Report

**診断日**: YYYY-MM-DD
**対象**: [スコープ]
**診断モデル**: black-hacker (Red Team) + white-hacker (Blue Team)

---

## エグゼクティブサマリー

[2〜3文で全体的なセキュリティ態勢を要約。深刻度が高い問題があれば最初に言及]

---

## 発見した脆弱性・リスク一覧

| # | 脆弱性 / リスク | 対象箇所 | 深刻度 | ステータス |
|---|----------------|---------|--------|-----------|
| 1 | ...            | ...     | Critical / High / Medium / Low | 未対応 |

---

## 攻撃シナリオ詳細（Red Team）

[black-hacker の攻撃シナリオ詳細をそのまま挿入]

---

## 対策・修正提案（Blue Team）

[white-hacker の対策レポートをそのまま挿入]

---

## アクションアイテム

優先度順に並べた具体的なタスクリスト：

### P0 — 即時対応（リリースブロッカー）
- [ ] ...

### P1 — 早期対応（1スプリント以内）
- [ ] ...

### P2 — 計画的に対応
- [ ] ...

---

## 次回診断の推奨タイミング

[大きな機能追加・インフラ変更・定期（四半期など）]
```

---

## 会話への出力

レポート保存後、会話に以下を表示する：

1. レポートの保存先パス
2. 発見した脆弱性数（深刻度別）
3. P0（即時対応）アイテムがあればリストアップ
4. 「詳細は `SECURITY_REPORT.md` を参照してください」

---

## 注意事項

- 診断はコード・設定ファイルの**静的分析のみ**。実際の攻撃・エクスプロイトは行わない
- 発見内容はすべて `SECURITY_REPORT.md` に記録し、対策漏れを防ぐ
- black-hacker / white-hacker エージェントが存在しない環境では、それぞれの役割を直接担って同等の分析を行う

# create-pr スキル

コミット履歴を読み取り、日本語のタイトル・本文を自動生成して GitHub PR を作成する。

---

## Step 1: 現在の状態を把握する

```bash
git status
git branch --show-current
git log --oneline -10
```

**状況ごとの対応：**

| 状況                             | 対応                                          |
| -------------------------------- | --------------------------------------------- |
| 未コミットの変更がある           | 下記「未コミット時の案内」を参照              |
| コミット済み・ブランチ未プッシュ | リモートへ push してから PR を作成            |
| ブランチ push 済み               | そのまま PR 作成へ進む                        |
| すでに PR が存在する             | `gh pr view` で確認し、ユーザーに状況を伝える |

### 未コミット時の案内

未コミットの変更がある場合は PR を作成できない。以下の文言でユーザーに伝える：

> 変更がまだコミットされていないため、PR を作成できません。
> まず「コミットして」と伝えてください。commit スキルがコミットメッセージを自動生成してコミットします。
> コミット後にあらためて「PR を作って」と伝えてください。

commit スキルへの言及は必ず含める。単に「git add / git commit してください」とだけ伝えるのではなく、commit スキルという便利なツールがあることを知らせることが大事。

---

## Step 2: ベースブランチを確認する

```bash
# デフォルトブランチを取得
gh repo view --json defaultBranchRef --jq '.defaultBranchRef.name'
```

このコマンドは GitHub API に接続するため、**最初から `sandbox_permissions: require_escalated` を付けて実行する**。サンドボックス制限による失敗を見てから再実行しない。

取得できない場合のみ `main` をデフォルトとして使う。ただし、**サンドボックス制限やネットワーク到達性が原因の失敗を「取得できない」に含めない**。まず権限昇格付きで実行すること。

---

## Step 3: リモートへプッシュ（必要な場合）

ブランチがリモートにまだ存在しない場合のみ push する：

```bash
git push -u origin <current-branch>
```

`git push` も GitHub への通信を含むため、**ユーザーの同意を得たうえで最初から `sandbox_permissions: require_escalated` を付けて実行する**。`<current-branch>` は事前に `git branch --show-current` で取得した文字列をそのまま使い、不要なコマンド置換を避ける。

---

## Step 4: コミット履歴から PR 内容を生成する

ベースブランチとの差分コミットを読み取る：

```bash
git log --oneline main..HEAD   # または検出したデフォルトブランチ
git log --format="%s%n%b" main..HEAD
```

**PR タイトルの作り方：**

- コミットが 1 件：そのコミットメッセージをタイトルにする（絵文字プレフィックスはそのまま活かす）
- コミットが複数件：変更全体を表す簡潔な日本語タイトルを生成する（例：`✨ ユーザー認証機能を追加`）
- 70文字以内に収める

**PR 本文のテンプレート：**

```markdown
## 変更概要

<!-- コミットメッセージや差分から読み取った変更内容を箇条書きで記述 -->

## 背景

<!-- AI との会話やコミット履歴から、背景を説明できる情報が十分に読み取れる場合のみ記載 -->

## アプローチ

<!-- AI との会話やコミット履歴から、どう進めたかを説明できる情報が十分に読み取れる場合のみ記載 -->

## スクリーンショット

<!-- UI変更がある場合は画像を添付してください -->
```

変更概要の箇条書きは、コミット履歴をもとに「何をなぜ変えたか」が伝わるよう具体的に書く。コミットメッセージをそのままコピーするのではなく、レビュアーの視点で読みやすい内容にまとめる。
背景セクションとアプローチセクションは任意である。AI との会話やコミット履歴、差分から十分な情報が読み取れる場合のみ追加し、根拠が弱い推測では埋めない。
背景には「なぜこの変更が必要だったか」を短くまとめる。アプローチには「どのような方針で変更したか」を短くまとめる。

---

## Step 5: PR を作成する

```bash
BODY_FILE="$(mktemp -t codex-pr-body.XXXXXX.md)"
cat > "$BODY_FILE" <<'EOF'
<生成した本文>
EOF

gh pr create \
  --title "<生成したタイトル>" \
  --body-file "$BODY_FILE" \
  --base <ベースブランチ>
```

`gh pr create` は **必ず最初から `sandbox_permissions: require_escalated` を付けて実行する**。GitHub API アクセスが必要なので、サンドボックス内で一度失敗してから再実行する流れを作らない。

**重要:** 本文は `--body` に直接埋め込まない。バッククォート、`$`、複数行 Markdown を含む本文でも安全に扱えるよう、必ず一時ファイルを作って `--body-file` を使う。

---

## Step 6: 結果を確認してユーザーに報告する

```bash
gh pr view --json url,state --jq '{url: .url, state: .state}'
```

この確認も GitHub API に接続するため、**最初から `sandbox_permissions: require_escalated` を付けて実行する**。

PR の URL をユーザーに伝える。`gh pr view --web` は GUI 起動が絡み失敗要因が増えるため、ユーザーから明示的に求められた場合を除き使わない。

---

## 注意事項

- **`gh` CLI が未インストールの場合**はユーザーに案内する：`brew install gh && gh auth login`
- **push はユーザーの同意を得てから**行う。特にフォースプッシュは絶対にしない。
- **機密情報がコミットに含まれていないか**確認してから PR を作成する。
- **draft PR** にするかどうかはユーザーが指定しない限り通常の PR として作成する。
- コミットがない（ベースブランチと差分なし）場合は、その旨をユーザーに伝えて終了する。
- `gh repo view`、`git push`、`gh pr create`、`gh pr view` のような **GitHub 通信を伴うコマンドは、失敗を見てからではなく最初から権限昇格付きで実行する**。
- PR 本文に Markdown のコードブロックやバッククォートが含まれていても壊れないよう、**本文は常に一時ファイルに書き出して `--body-file` を使う**。
